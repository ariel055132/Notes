## Watcher vs Cron
1. Why separate the watcher from the worker? 
* Watcher (detects / claim work)
* Worker (executes work)
* *Independent Scaling* 
    * Watcher loads scales with number of queues + polling / listing frequency
    * Worker loads scales with job runtime + CPU / memory needs
    * Keeping watcher and worker separate let user run few watchers, many workers (or vice versa) without overprovisioning
* *Isloation of failure modes*
    * Watchers and workers should not affect each other.
    * For example, when workers crash due to bad inputs, OOM. They should not break the work of watcher
    * If they do not separate, both of them will failure.
* *Security Boundary*
    * Watcher may need privileged credentials in order to find the work (in DB, queue management, broad read access)
    * Workers can run with least privilege (read the jobs from queue.)
    * Isolating can reduce blast radius
* *Cost efficiency*
    * Watchers are often network/IO bound (check queue / db for jobs), not compute heavy
    * However, workers may be compute heavy, but not network/IO bound
    * Therefore, we can run watchers on smaller instances, while workers on specialized nodes at the same time

2. What problems does a single cron job that both scans and executes have?
* *Scaling*
    * Cannot scale "finding work (watcher)" and "doing work (worker)" independently
    * One heavy work will block the next scan / finding work.
    * Or multiple crons will lead to multiple scans and race conditions on the same job execution
* Failure*
    * Execution can be killed by a bug or OOM
    * If job will stop scan if the execution is killed when they are combined together
* *Work delay*
    * As cron is time-based / time-sliced. (scheduling by time)
    * If scan + executes task takes longer time than the time stated in crontab, the next run queue will behind or skips
* *Security*
    * Broader execution / query permission may need if a cron job can both scans and executes.
    * Harder to apply least privilege

## Time Bucket Partitioning
1. Instead of `SELECT * WHERE scheduled_at <= now()`, why partition jobs by time bucket (e.g., hour)?
* Avoid full table scan
* *Bounded work per poll*
    * 
* *Database partition pruning*
    * If database is physically parition by time, queries filter by bucket =:hour can prune old parition.
    * User never scan cold data
* *Paralleism*
    * Workers can take different bucket and claim jobs, instead of everyone fighting over the same condition scheduled_at <= now()
* *Clearer Operations*
    * More easier to redo the operation at the specified time bucket
2. What happens to query performance at 1M+ jobs without partitioning?
* *Indexes and I/O get heavier*
    * Without paritioning, that means when we use index for scheduling grows with all rows (larger indexes)
    * More pages to traverse for range scans
    * More disk reads under concurrent load

## Tool Naming
1. Why `task.create` instead of `createTask`?
* Why naming need to use "resource + action" pattern instead of camelCase.
* *Few ambiguous verbs*
    * create alone is meaningless
    * task.create reads like create task
    * More clearer as less cramming English into function name
* *Clear namespace grouping*
    * task.create, task.update, task.delete natually group together in operations in task / tool-list
* *Avoid name collisions*
    * In larger servers or multi-server setups, createTask may appear multiple times (collide with other domains), while task.create is more clear
2. How does naming convention affect LLM tool selection accuracy?
* Strong verb-object clarity improves intent match
    * Names like task.create, user.getProfile, logs.search map directly to common user requests, so the model picks the right tool faster 
    * resoruce + action
* Using consistent pattern to improve generalization
    * If tool follows one pattern, the model can learn local grammar, then predicts the right sibling action more accuractely

## Registry vs If-Else
1. Why use a dictionary registry to route tool calls instead of if-else chains?
* O(1) lookup (dictionary <-> map) and cleaner dispatch
* Easier to scale as adding a line to registry
2. What happens when you need to add the 20th tool?
* *More fragile code*
    * Need to edit a long conditional block if add the 20th tool to if-else chains
    * Increasing chance of wrong order, copy-paste bugs, or missing returns

### Example (dictionary registry) & (if-else chains)
```python
def get_weather(args):
    return f"Forecast for {args.get('city', 'unknown')}"

def get_alerts(args):
    return f"Alerts for state {args.get('state', 'unknown')}"

def create_ticket(args):
    return f"Ticket created: {args.get('title', 'untitled')}"


# 1) If-else chain router
def route_with_if_else(tool_name, args):
    if tool_name == "weather.getForecast":
        return get_weather(args)
    elif tool_name == "weather.getAlerts":
        return get_alerts(args)
    elif tool_name == "ticket.create":
        return create_ticket(args)
    else:
        raise ValueError(f"Unknown tool: {tool_name}")


# 2) Dictionary registry router
TOOL_REGISTRY = {
    "weather.getForecast": get_weather,
    "weather.getAlerts": get_alerts,
    "ticket.create": create_ticket,
}

def route_with_registry(tool_name, args):
    handler = TOOL_REGISTRY.get(tool_name)
    if handler is None:
        raise ValueError(f"Unknown tool: {tool_name}")
    return handler(args)

# Demo
print(route_with_if_else("weather.getForecast", {"city": "New York"}))
print(route_with_registry("weather.getForecast", {"city": "New York"}))
```
