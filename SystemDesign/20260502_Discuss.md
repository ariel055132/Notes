## Static vs Dynamic QR Code
1. Why does this system use dynamic QR codes (encode short URL) instead of static (encode original URL directly)?
* Static (靜態)
    * QR code stores the final destination URL directly
    * Once printed, destination URL cannot be changed.
    * No built-in backend tracking unless the destination site itself tracks visits
* Dynamic (動態)
    * QR code stores a short managed URL that redirects to the real destination url
    * Destination url can be changed later without reprinting
    * Easy to add analytics, expiration...
* As the system requirments states that, users can add, modify, and delete QR codes. Therefore, static is not suitable to the case as it is impossible to modify after creation.
* Need to use dynamic url instead of static url
2. When would you choose static instead?
* For something that never changed / seldomly changed (E.G: Enterprise website, Product website)
* Offline environments
* Privacy-sensitive contents 

## Token Generation
1. How will you generate short URL tokens?
    * Hash + Base 62 encoder
2. What happens when two different URLs produce the same token?
    1. Retry
    2. Check whether the new token still exist in the Database
    3. If still exist, repeat step 1
    4. If not exist, save the token to the database
    5. If all retries fail, raise Runtime error
3. How does collision probability change as the number of tokens grows?
    * Assumption: Base 62 is used, and length 7 token is generated
    * Token Space: 62^7
    * If n = 1000, about 0.000014%
    * If n = 1000000, about 0.014%
    * If n = 1000000000, about 12%

## Redirect Strategy
1. Why 302 (temporary) instead of 301 (permanent)?
* 302 (temporary)
    * This page has *temporarily moved*.
    * No caching. Every Scan hits server / DB first
* 301 (permanent)
    * The page has *permanently moved*
    * The browser caches the redirect indefinitely 
    * When user visits again, it goes directly to the destination URL with cache. Server never contacted again.
* As system requirements states that the url will be changed and saved to server. Therefore, if server was not contacted, cannot obtained the correct url.
2. What are the trade-offs for analytics, URL modification, and latency?
* Analytics
    * 302: Every scan is logged
    * 301: Cached scans bypass servers (except the first time), thus scan count is fixed after first visits
* URL modification
    * 302: Next scan will fetch the new destination URL from server
    * 301: As old URl is cached and bypass server, the update will be ignored entirely
* Latency
    * 302: Some latency will exist, as it needs redirection to server before reaching to URL.
    * 301: Zero-Latency (will not redirect to server, bypass server)

## URL Normalization
1. What normalization rules do you need?
* QR Code system should not create duplicate tokens just because the URL was typed differently.

2. Why is http://Example.com/ and https://example.com potentially the same URL?
* According to RFC3986, url should use lowercase letters. 
* Therefore, Example === example in url. 

## Error Semantics
* Different error codes should be used for several situations
* Tokens does not exist in DB, return 404 (Not Found)
* Tokens exists in DB && not deleted, return 302 (Found)
* Token exists in DB && deleted, return 410 (Gone)
* Ref: https://developer.mozilla.org/zh-TW/docs/Web/HTTP/Reference/Status
